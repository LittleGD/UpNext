## 문제 & 제약

### 기존 웰니스 앱의 구조적 한계

대부분의 습관 앱은 동일한 패턴을 따른다: 할 일 목록을 만들고, 체크하고, 반복한다. 나 스스로가 Habitica, Fabulous, 루티너리 등 3개의 습관 앱을 사용해봤지만, 평균 11일 만에 포기했다. 이유는 세 가지로 수렴했다.

- **결정 피로 (Decision Fatigue)**: "오늘 뭘 해야 하지?"를 매일 고민해야 한다
- **루틴의 권태**: 같은 할 일이 반복되면 2주차부터 앱을 여는 빈도가 급감한다
- **실패의 부담**: 하루를 놓치면 "어차피 망했다"는 심리가 작동하고, 다시 돌아오기 어렵다

### UpNext의 해법: 랜덤성 + 게임 메커니즘

로그라이크 게임에서 영감을 받았다. 로그라이크의 핵심은 **매 회차가 다르다**는 점이다. 카드 드로우라는 랜덤 요소를 넣으면:

- 선택이 아니라 **발견**이 된다 -- "오늘은 이걸 뽑았네!"
- 매일 조합이 달라져 **권태가 줄어든다**
- 하루를 놓쳐도 내일 새 카드가 나오므로 **재시작 부담이 없다**

> "오늘 뭐 하지?"를 "오늘 뭐 뽑았지?"로 바꾸는 것이 핵심 경험이다.

### 프로젝트 제약 조건

이 프로젝트는 처음부터 명확한 제약 안에서 시작했다:

- **시간**: 2일. 과도한 스코프는 불가능. 핵심 루프(드로우 → 선택 → 완료)에 집중해야 했다
- **인원**: 1인. 백엔드 구축에 시간을 쓸 수 없어 오프라인 우선(localStorage) + 옵셔널 클라우드 싱크 구조를 선택
- **기술 경험**: UX/UI 디자이너 배경. CSS/HTML 기본 지식은 있지만 프레임워크 경험은 이번이 처음. AI 협업으로 학습과 구현을 동시에 진행해야 했다

이 제약들이 모든 기술 결정의 방향을 결정했다. "2일 안에 1인이 완성할 수 있는가?"라는 필터가 매 선택에 적용되었다.

## 핵심 디자인 결정

### 결정 1: 다크 + 네온 컬러 시스템

![컬러 시스템 — 5가지 액센트 컬러와 다크 서피스 팔레트](screenshots/color-system-swatches.png)

**고려한 대안:**

| 방향 | 장점 | 단점 |
|------|------|------|
| **라이트 테마 + 파스텔** | 웰니스 앱의 전형적 톤. 친근함 | 게임 정체성이 사라짐. "또 하나의 습관 앱"이 됨 |
| **다크 + 모노크롬** | 세련됨 | 레어리티 시스템의 시각적 보상 효과가 약함 |
| **다크 + 네온 (선택)** | 게임 정체성 강화. 레어리티별 색상 매핑으로 즉각적 피드백 | 접근성 주의 필요. 네온 과용 시 피로감 |

**왜 이 방향인가:** UpNext의 핵심 경험은 "카드를 뽑는 순간의 기대감"이다. 라이트 테마에서는 이 감각이 전달되지 않았다. 네온 컬러를 레어리티 등급에 직접 매핑하면, 카드를 뽑는 순간 **색상만으로** "좋은 카드를 뽑았다"는 감각적 피드백이 즉각 전달된다.

| 등급 | 컬러 | HEX | 용도 |
|------|-------|-----|------|
| Normal | Gray | `#808080` | 기본 등급. 차분하게 |
| Rare | Aquamarine | `#9BF0E1` | 눈에 들어오기 시작하는 지점 |
| Unique | Fushia | `#F037A5` | 확실한 시각적 보상 |
| Legend | Citric | `#CDF564` | 가장 강한 시각적 자극. 희소성 강조 |

![레어리티별 카드 비교 — Normal부터 Legend까지 실제 앱 내 카드 모습](screenshots/rarity-comparison.png)

**디자인-코드 연결:** 이 컬러 시스템은 CSS 변수로 `globals.css`에 토큰화되어 있다. `--rarity-normal`, `--rarity-rare`, `--rarity-unique`, `--rarity-legend`로 정의하고, Tailwind의 `@theme inline`으로 연결하여 `text-rarity-legend` 같은 유틸리티 클래스로 사용한다. 디자인 토큰이 곧 코드다.

```css
/* globals.css — 디자인 토큰 */
:root {
  --rarity-normal: #808080;
  --rarity-rare: #9BF0E1;
  --rarity-unique: #F037A5;
  --rarity-legend: #CDF564;
}

@theme inline {
  --color-rarity-normal: var(--rarity-normal);
  --color-rarity-rare: var(--rarity-rare);
  --color-rarity-unique: var(--rarity-unique);
  --color-rarity-legend: var(--rarity-legend);
}
```

**트레이드오프:** 네온 컬러를 동적으로 적용하다 보니, Tailwind의 프로덕션 빌드에서 CSS purge 이슈가 발생했다. `bg-${color}` 같은 동적 클래스가 빌드 타임에 감지되지 않아 제거된 것이다. 이를 인라인 스타일로 전환하여 해결했다. 디자인 결정이 기술적 문제를 만들고, 그 문제를 기술적으로 해결한 사례다.

### 결정 2: 언어별 폰트 스위칭

![타이포그래피 비교 — 4개 언어에서 동일한 화면이 각기 다른 폰트로 보이는 모습](screenshots/typography-4-languages.png)

**고려한 대안:**

| 방향 | 장점 | 단점 |
|------|------|------|
| **Noto Sans 통일** | 안전. 모든 언어 커버. 로딩 최적화 | 게임 정체성 사라짐. 픽셀 감성 전무 |
| **픽셀 폰트 하나로 통일** | 일관성 | CJK 지원하는 좋은 픽셀 폰트가 없음. 가독성 위험 |
| **언어별 맞춤 폰트 (선택)** | 각 언어의 문화적 뉘앙스 반영. 픽셀/레트로 감성 유지 | 폰트 로딩 증가. 구현 복잡도 증가 |

**왜 이 방향인가:** UpNext의 톤은 "레트로 게임"이다. Noto Sans는 안전하지만, 게임을 켠 느낌을 주지 못한다. 한국어에는 April16Promise(픽셀 감성), 영어에는 slight-chance-mono(레트로 모노스페이스), 일본어에는 WDXL Lubrifont(부드러운 둥근체), 중국어에는 ZCOOL QingKe HuangYou(손글씨 캐릭터)를 선택했다.

**구현 핵심:** JavaScript 없이 CSS만으로 처리했다. `html[lang]` 선택자로 CSS 변수를 오버라이드하는 방식이다:

```css
:root { --font-display: 'April16Promise', sans-serif; }
html[lang="en"] { --font-display: 'slight-chance-mono', sans-serif; }
html[lang="ja"] { --font-display: 'WDXL Lubrifont JP N', sans-serif; }
html[lang="zh"] { --font-display: 'ZCOOL QingKe HuangYou', sans-serif; }
```

언어를 바꾸면 폰트가 자연스럽게 전환된다. JS 런타임 비용 없이 CSS Cascade만으로 동작하므로 성능 부담이 없다.

**트레이드오프:** 4개 폰트 파일을 로드하면 초기 로딩이 느려질 수 있다. `font-display: swap`으로 FOUT(Flash of Unstyled Text)를 허용하되, CDN(jsDelivr)에서 woff2만 로드하여 용량을 최소화했다. 완벽한 해결은 아니지만, 게임 톤 유지라는 우선순위에서 수용 가능한 트레이드오프라고 판단했다.

### 결정 3: Zustand + localStorage 오프라인 우선

**고려한 대안:**

| 방향 | 장점 | 단점 |
|------|------|------|
| **Redux** | 생태계 크고, 미들웨어 풍부 | 보일러플레이트 과다. 2일 안에 학습+구현 비현실적 |
| **React Context API** | 내장, 추가 설치 없음 | 상태 변경마다 전체 리렌더링. 게임 상태처럼 자주 변하는 데이터에 부적합 |
| **Zustand (선택)** | 보일러플레이트 최소. 셀렉터 기반 리렌더링 최적화 | 생태계가 Redux보다 작음 |

**왜 이 조합인가:** 게임 상태는 "오늘 뽑은 카드, 선택한 카드, 완료 여부, XP, 레벨, 스트릭"처럼 자주 변하고 서로 연결된 데이터다. Context API를 쓰면 카드 하나를 선택할 때마다 관련 없는 UI까지 리렌더링된다. Zustand는 셀렉터(`useGameStore(s => s.daily)`)로 필요한 슬라이스만 구독하여 이 문제를 해결한다.

localStorage를 기본 영속화 레이어로, Firebase Firestore를 옵셔널 클라우드 싱크로 설계한 이유: **1인 프로젝트에서 백엔드 장애는 곧 서비스 중단**이다. 네트워크 없이도 100% 기능이 동작하면, 인프라 리스크를 제거할 수 있다. 로그인하면 기기 간 동기화가 추가되는 구조로, 최소 기능과 확장 기능을 분리했다.

### 디자인 규칙: 왜 보더 없는 다크 UI인가

프로젝트를 진행하며 세 가지 디자인 규칙을 확립하고, Claude의 persistent memory에 등록하여 세션이 바뀌어도 일관성을 유지했다.

| 규칙 | 이유 |
|------|------|
| 아이콘에 배경 박스를 씌우지 않는다 | pixelarticons의 픽셀 캐릭터가 박스 안에 갇히면 촌스러워진다. 아이콘 자체의 실루엣이 인식 단위 |
| 카드와 버튼에 보더를 넣지 않는다 | 다크 배경 + 네온 글로우 환경에서 보더는 시각적 노이즈. 배경색 차이(`#141414` vs `#1E1E1E`)로 계층 구분 |
| 배경색 차이와 글로우 이펙트로 계층을 구분한다 | 3단계 다크 서피스(`--bg-primary` → `--bg-surface` → `--bg-elevated`)가 보더 역할을 대체 |

처음에는 카드에 보더를 넣어봤다. 다크 배경 위의 네온 보더가 과하게 느껴졌고, 68장의 카드가 한 화면에 나열될 때 보더가 시각적 노이즈를 만들었다. 보더를 제거하고 배경색 차이만으로 구분하니, 카드의 **내용(레어리티 컬러, 카테고리 아이콘)**에 시선이 집중되었다.

## 솔루션 & 구현 하이라이트

### 핵심 루프: 드로우 → 선택 → 완료

![카드 드로우 플로우 — 6장 드로우에서 선택, 완료까지의 흐름](screenshots/card-flow-gif.gif)

1. 매일 덱에서 **6장의 챌린지 카드**를 랜덤 드로우
2. 난이도 모드에 따라 **1~3장 선택** (일반 / 갓생 / 초갓생)
3. 미션 수행 후 완료 체크 -- XP 획득, 레벨업, 새 카드 해금
4. **68장의 챌린지 카드**, 7개 카테고리, 4단계 레어리티

### 가중 카드 드로우 알고리즘

단순 랜덤이 아니라, 레어리티에 가중치를 부여한 확률 시스템을 구현했다.

| 등급 | 가중치 | 확률 | 디자인 의도 |
|------|--------|------|------------|
| Normal | 60 | 60% | 기본 경험. 부담 없이 수행 가능한 미션 |
| Rare | 25 | 25% | 눈에 띄기 시작. "오, 좀 다른 게 나왔네" |
| Unique | 12 | 12% | 확실한 보상감. 도전적인 미션 |
| Legend | 3 | 3% | 희소성이 흥분을 만듦. 뽑는 순간의 쾌감 |

이 비율은 로그라이크 게임(Slay the Spire, Balatro)의 드롭률 패턴을 참고했다. 핵심 원리는: Normal만 나오면 재미없고, Legend가 너무 자주 나오면 희소성이 사라진다. **6장 중 평균 1~2장이 Rare 이상**이 되는 지점에서 "오늘은 운이 좋다!"라는 감각이 발생한다.

여기에 두 가지 보정 로직을 추가했다:

**카테고리 다양성:** 처음 4장은 이미 뽑힌 카테고리를 피한다. 운동 카드만 4장 나오는 상황을 방지.

**피티 시스템 (Pity System):** 6장이 모두 Normal이면, 마지막 1장을 Rare 이상으로 강제 교체. 가챠 게임에서 차용한 메커니즘으로, "오늘은 재미없다"는 경험을 없앤다.

```typescript
// 피티 시스템: 6장이 모두 normal이면 하나를 교체
const hasNonNormal = drawn.some((card) => card.rarity !== "normal");
if (!hasNonNormal) {
  const nonNormalCards = unlockedCards.filter(
    (card) => card.rarity !== "normal" && !usedIds.has(card.id)
  );
  if (nonNormalCards.length > 0) {
    const replacement = nonNormalCards[Math.floor(Math.random() * nonNormalCards.length)];
    drawn[drawn.length - 1] = replacement;
  }
}
```

### 모션 시스템: 과하지 않되, 반응이 있는

카드 게임에서 피드백은 곧 보상이다. 카드를 뽑았을 때, 선택했을 때, 완료했을 때 -- 각 순간에 적절한 물리적 반응이 있어야 "게임을 하고 있다"는 감각이 유지된다. 모션이 과하면 산만해지고, 없으면 밋밋하다. "눌렀을 때 반응이 온다"는 최소한의 촉감을 일관되게 유지하는 것이 원칙이었다.

Framer Motion을 선택한 이유: CSS 애니메이션도 고려했지만, `layout` 애니메이션(카드 선택 시 위치 자동 전환)은 CSS만으로 구현하기 어렵다. Framer Motion은 선언적 API로 `layout` prop 하나면 요소의 위치 변화를 자동으로 애니메이트한다. spring 물리 기반 모션이 기계적이지 않은 자연스러운 느낌을 준다.

`motion.ts`에 프로젝트 전체에서 공유하는 모션 프리셋을 정의했다:

```typescript
// 스프링 프리셋 — 용도별로 느낌이 다름
export const springSnappy: Transition = { type: "spring", stiffness: 500, damping: 30 };
export const springGentle: Transition = { type: "spring", stiffness: 200, damping: 20 };
export const springBouncy: Transition = { type: "spring", stiffness: 300, damping: 15 };

// 카드 등장 — opacity: 0, y: 20에서 자연스럽게 슬라이드 인
export const fadeInUp: Variants = {
  hidden: { opacity: 0, y: 20 },
  visible: { opacity: 1, y: 0 },
};

// 완료 체크 — 스탬프를 찍듯 scale 2에서 1로 착지
export const stampIn: Variants = {
  hidden: { scale: 2, opacity: 0 },
  visible: { scale: 1, opacity: 1,
    transition: { type: "spring", stiffness: 400, damping: 15 } },
};
```

완료 체크에 stampIn을 적용한 이유: 체크 아이콘이 크게 나타났다가 제자리에 착지하는 모션은 "스탬프를 찍는" 물리적 무게감을 전달한다. 작은 도전을 끝냈을 때 "해냈다!"라는 달성감을 시각적으로 강화하기 위한 선택이다. stiffness 400 / damping 15 조합은 빠르게 착지하되 약간의 바운스가 남아 경쾌함을 더한다.

`scale: 0.95`(버튼 프레스)라는 값은 0.9, 0.95, 0.98을 비교 테스트한 결과다. 0.9는 너무 과장되어 의도하지 않은 "불안감"을, 0.98은 거의 느껴지지 않는 수준이었다. 0.95가 "눌렸다"는 촉감을 주면서도 과하지 않은 지점이었다.

### 오로라 배경: 살아있는 화면

다크 화면이 단순한 빈 공간이 아니라 "게임 세계"로 느껴지려면, 화면 자체가 살아 숨쉬어야 한다. 오로라는 그 숨결이다. 정적인 다크 배경에서는 카드만 동동 떠 있는 느낌이었지만, 배경이 느리게 움직이자 화면 전체가 하나의 공간으로 연결되었다.

![오로라 배경 효과 — 화면 하단에 은은하게 흐르는 보라색 그라디언트](screenshots/aurora-effect.gif)

세 개의 CSS 레이어를 겹쳐 깊이감을 만들었다:
- **레이어 1** (딥 퍼플 베이스): `rgba(59, 7, 100, 0.45)` -- 하단 넓게, 부드러운 확산
- **레이어 2** (메인 오로라 밴드): 보라/마젠타 물결 -- `aurora-drift-alt` 25초 주기
- **레이어 3** (상단 엣지 하이라이트): `rgba(160, 120, 230, 0.1)` -- 미세한 빛

`aurora-drift`와 `aurora-breathe` 키프레임으로 각 레이어가 서로 다른 속도와 방향으로 움직인다. `pointer-events: none`으로 인터랙션을 방해하지 않으면서, 화면이 하나의 공간으로 느껴지게 한다.

### 문제 해결: 실전에서 배운 것들

#### CSS Purge -- 비주얼 이펙트가 프로덕션에서 사라지다

![Before/After — 로컬에서는 보이던 오로라가 프로덕션에서 사라진 모습](screenshots/css-purge-before-after.png)

**증상**: 로컬에서는 보이던 오로라 배경과 픽셀 별이 Vercel 배포 후 사라짐.

**원인**: Tailwind CSS 4의 프로덕션 빌드가 동적으로 조합된 클래스(`bg-${color}`)를 감지하지 못해 제거함.

**해결**: 동적 Tailwind 클래스를 모두 인라인 스타일로 전환. CSS purge의 영향을 받지 않는 안전한 방식.

**교훈**: Tailwind의 동적 클래스는 프로덕션에서 위험하다. 런타임에 결정되는 스타일은 인라인으로 처리하는 것이 원칙이다. 이 경험이 앞서 서술한 컬러 시스템 설계에 역으로 영향을 미쳤다 -- 레어리티 컬러를 CSS 변수로 토큰화한 이유 중 하나가 purge 안정성이었다.

#### localStorage 잔존 데이터 -- 카드 이름이 번역되지 않다

**증상**: 4개 국어 번역을 추가한 후에도, 기존에 뽑아둔 카드의 제목이 한국어로만 표시됨.

**원인**: 카드를 드로우하면 카드 객체 전체가 localStorage에 저장된다. 번역 필드를 추가하기 전에 저장된 데이터에는 해당 필드가 없었다. 앱이 옛날 데이터를 그대로 사용하니, 번역이 동작하지 않았다.

**해결**: i18n 모듈에 `resolveCard()` 함수를 추가. 저장된 데이터에서 ID만 읽어 최신 카드 데이터를 `ALL_CARDS` 맵에서 조회하도록 변경했다. 저장 데이터는 "어떤 카드를 뽑았는지"의 참조(ID)로만 사용하고, 표시 데이터는 항상 최신 소스에서 가져온다.

```typescript
// i18n/index.ts — 영속화된 데이터의 스키마 변경에 대응
let _cardMap: Map<string, ChallengeCard> | null = null;
function getCardMap(): Map<string, ChallengeCard> {
  if (!_cardMap) _cardMap = new Map(ALL_CARDS.map((c) => [c.id, c]));
  return _cardMap;
}
function resolveCard(card: ChallengeCard): ChallengeCard {
  return getCardMap().get(card.id) ?? card;
}
```

**교훈**: 영속화된 데이터의 스키마는 반드시 변한다. 데이터 자체를 저장하기보다 **참조(ID)를 저장하고 런타임에 resolve**하는 패턴이 안전하다.

#### Lighthouse 60점 -- 성능 병목을 진단하고 체계적으로 개선하다

**증상**: PageSpeed Insights 모바일 측정에서 Performance 60점. FCP 6.4초, LCP 7.3초.

**진단 과정**: Lighthouse의 "Eliminate render-blocking resources" 항목에서 외부 폰트 CSS 4개가 FCP를 ~3.5초 지연시키고 있었다. 4개 언어 폰트가 현재 언어와 무관하게 **모두** 렌더 블로킹으로 로딩되고 있었다. 추가로 Firebase SDK가 로그인 여부와 관계없이 초기 번들에 포함되어 ~250KiB를 차지했다.

**해결**: "가장 큰 병목 하나부터 제거한다"는 원칙으로, 4단계로 체계적으로 접근했다:

1. **폰트 셀프호스팅**: 외부 CDN → `next/font/local` + `next/font/google`로 전환. 빌드 타임에 폰트를 다운로드하여 같은 도메인에서 서빙
2. **Firebase 지연 로딩**: 정적 `import` → `async getFirebase()` 동적 싱글턴. 로그인 시점에만 로딩
3. **이펙트 지연**: 오로라 + 별 → `dynamic({ ssr: false })` + `requestIdleCallback`
4. **코드 스플릿**: 조건부 컴포넌트 3개 → `next/dynamic`

**교훈**: 성능 최적화는 "느리다"에서 시작하지 않는다. **"왜 느린가?"를 측정**하는 것에서 시작한다. 감이 아니라 데이터로 병목을 찾으면, 전체의 70%를 차지하는 단일 원인에 집중할 수 있다. 이 경험을 통해 Lighthouse를 읽고 해석하는 능력, Next.js의 폰트 최적화 시스템, 동적 import 패턴을 실전에서 배웠다.

#### 병렬 에이전트 머지 -- 두 워크트리가 같은 파일을 수정하다

**증상**: 일본어 번역 에이전트와 중국어 번역 에이전트가 동시에 `cards.ts`를 수정. 단순 머지 불가.

**해결**: 양쪽 워크트리의 결과물을 직접 읽어서, JA 필드와 ZH 필드를 수동으로 합쳐 하나의 커밋으로 통합. 별도의 QA 에이전트로 68장 x 2개 언어가 모두 반영되었는지 검증.

**교훈**: 병렬 AI 에이전트는 파일 단위가 아니라 **필드 단위**로 충돌할 수 있다. 작업을 파일로 분리하거나, 순차 처리하는 것이 더 안전할 수 있다.

## AI 협업

이 프로젝트는 "AI가 코드를 짜줬다"가 아니라 "AI와 함께 설계하고 배웠다"에 가깝다.

### 역할 분담 매트릭스

| 영역 | 나의 기여 (Human) | AI 역할 (Claude) | 왜 이 분담이 효과적이었나 |
|------|-------------------|-----------------|------------------------|
| **컨셉/기획** | 로그라이크 메커니즘 설계, 카드 카테고리 정의, 레어리티 확률 설계 | -- | 게임 경험에 대한 직관은 AI가 대체 불가 |
| **비주얼 디자인** | 컬러 시스템, 디자인 규칙, 레이아웃 결정 | -- | 디자인 판단은 100% 나의 영역 |
| **프론트엔드 구현** | 아키텍처 결정, 코드 리뷰, 모든 디버깅 | 코드 생성 가속, 구현 패턴 제안 | 내가 설계하고, AI가 빠르게 번역 |
| **번역** | 품질 QA, 머지 전략 수립 | 4개 국어 텍스트 생성 (68장 x 4언어) | 번역 속도와 규모에서 AI가 압도적 |
| **문제 진단** | 증상 관찰, 원인 추론, 해결 방향 결정 | 해결 코드 제안 | 진단 능력은 AI를 통해 학습한 핵심 역량 |

### 원칙: "무엇을" 뿐 아니라 "왜"를 묻는다

모든 코드 결정에 대해 이유를 설명받았다:

- Fisher-Yates Shuffle을 왜 쓰는가? → 균등 분포 보장
- Zustand를 왜 선택했는가? → Context API 대비 리렌더링 최적화, Redux 대비 보일러플레이트 감소
- `font-display: swap`을 왜 쓰는가? → FOUT를 허용하되 렌더 블로킹을 방지

이 "왜"들이 쌓여서 기술적 판단력이 되었다. AI는 답을 줬지만, **"이 답이 맞는가?"를 판단하는 것은 나의 몫**이었다.

### AI가 실패한 순간들

모든 AI 출력물이 바로 사용 가능했던 것은 아니다:

- **톤 앤 보이스**: Claude가 생성한 초기 카드 텍스트는 "하루에 물 8잔 마시기" 같은 평범한 문장이었다. 로그라이크 게임의 톤에 맞지 않았다. "RPG 퀘스트 설명처럼 써달라"고 프롬프트를 수정한 끝에야 원하는 톤을 잡았다.
- **오로라 CSS**: 처음 3번의 결과물은 단순한 선형 그라디언트였다. 내가 직접 Codepen에서 참고 사례를 찾아 프롬프트에 첨부한 후에야 원하는 수준의 다층 레이어 오로라 코드를 얻었다. **레퍼런스를 제시하는 것 자체가 디자이너의 역량**이라는 것을 깨달았다.
- **설정 페이지 레이아웃**: AI가 처음 생성한 설정 페이지는 iOS 스타일의 단순 리스트였다. 내가 원한 것은 카드형 그룹 레이아웃이었다. 최종 결과물을 얻기까지 3번의 이터레이션이 필요했다.

### 효율 → 깊이 전환

AI 코드 생성으로 절약한 시간을 어디에 재투자했는가:

- **마이크로인터랙션 튜닝**: `scale` 값 비교 테스트, spring 파라미터 조정, 스태거 타이밍 미세 조정
- **4개 국어 번역 QA**: 68장 x 4언어 = 272개 텍스트의 맥락 적절성 검수
- **디버깅 학습**: 에러를 단순히 "고쳐달라"고 하지 않고, 원인을 이해한 후 해결 코드를 요청하는 방식으로 전환

### 학습 문서화: Obsidian 피드백 루프

매 세션에서 배운 내용을 Obsidian 호환 마크다운으로 정리했다:

- 무엇을 요청했는가 → 무엇이 전달되었는가 → 어떤 교훈이 있었는가 → 다음에 더 잘 요청하려면?

이 피드백 루프가 AI 협업의 품질을 세션마다 높여갔다. 디자인 규칙을 persistent memory에 등록한 것도 이 문서화 습관에서 나왔다 -- "이 규칙을 매번 말해야 하는가?" → "아니, 영속화하자."

#### 병목 분석 — 1차: 폰트 렌더 블로킹

Performance 60의 가장 큰 원인은 **외부 CDN 폰트 4종이 언어 무관하게 렌더 블로킹으로 동시 로딩**되는 것이었다. 추가로 Firebase SDK ~250KiB가 무조건 로딩되고, 시각 이펙트가 초기 GPU를 경쟁했다.

| 병목 | 영향 | 원인 |
|------|------|------|
| 외부 폰트 CDN 4종 | FCP +3~4s | jsdelivr, Google Fonts, Typekit이 렌더 블로킹 |
| Firebase SDK | 초기 JS +250KiB | 로그인 여부 무관 정적 import |
| 오로라 + 별 이펙트 | FCP +0.3s | SSR + 즉시 마운트로 초기 GPU 블러 경쟁 |

#### 최적화 Phase 1-3: 폰트 + Firebase + 이펙트

**Phase 1 — 폰트 셀프호스팅 (FCP 6.4s → 3.2s)**

외부 CDN 의존을 제거하고 Next.js 빌드타임 셀프호스팅으로 전환했다:

| 폰트 | Before | After |
|-------|--------|-------|
| April16Promise (한국어) | jsdelivr CDN `@font-face` | `next/font/local` — woff2 로컬 번들, `font-display: optional` |
| WDXL Lubrifont JP N (일본어) | Google Fonts `<link>` 렌더 블로킹 | LanguageSync에서 언어 전환 시점에 동적 주입 |
| ZCOOL QingKe HuangYou (중국어) | Google Fonts `<link>` 렌더 블로킹 | LanguageSync에서 언어 전환 시점에 동적 주입 |
| slight-chance-mono (영어) | Typekit `<link rel="stylesheet">` 렌더 블로킹 | `next/script` `afterInteractive` 비동기 주입 |

CJK 폰트를 `next/font/google`으로 전환하는 것도 시도했지만, `preload: false`여도 수십 개의 unicode-range @font-face 규칙이 메인 CSS 번들에 포함되어 렌더 블로킹을 330ms까지 비대화시켰다. 결국 **언어 전환 시점에 Google Fonts CSS를 동적 주입**하는 방식이 가장 효과적이었다.

**Phase 2 — Firebase SDK 지연 로딩 (초기 JS -250KiB)**

정적 `import` 3줄을 `async getFirebase()` 싱글턴 패턴으로 전환. `requestIdleCallback`으로 FCP/LCP 이후 idle 시점에 SDK를 파싱하여 TBT도 감소시켰다.

```typescript
// Before: 모든 사용자가 ~250KiB를 즉시 로딩
import { initializeApp } from "firebase/app";

// After: idle 시점에 동적 로딩
whenIdle(async () => {
  const { auth } = await getFirebase();
  // ...
});
```

**Phase 3 — 시각 이펙트 코드 스플릿**

오로라 배경과 별 캔버스를 `next/dynamic({ ssr: false })`로 전환. 서버에서 렌더링하지 않고 클라이언트에서만 로딩되어 초기 HTML 크기와 하이드레이션 비용이 감소했다.

#### 병목 분석 — 2차: JS 하이드레이션

Phase 1-3으로 FCP는 6.4s → 3.2s로 대폭 개선되었지만, **LCP가 오히려 8.5s로 악화**되는 현상이 발생했다. 측정과 분석을 반복한 끝에 근본 원인을 발견했다:

> 원래 버전에서 폰트 렌더 블로킹(6.4s)이 JS 파싱 시간을 "숨겨주고" 있었다. 폰트 블로킹을 제거하자 **JS 하이드레이션 병목이 드러난 것**이다.

```
Before: 폰트 블로킹 6.4s (그 동안 JS 조용히 파싱) → 모든 것이 함께 렌더 → LCP 7.3s
After:  FCP 3.2s (skeleton) → JS 다운로드 2.5s → SplashScreen 2.5s 타이머 → LCP 8.5s
```

Lighthouse의 LCP 요소는 온보딩 Description 화면의 텍스트였다. 이 텍스트가 렌더되려면:
1. JS 번들 다운로드 + 파싱 (모바일 4x CPU throttle에서 ~2.5s)
2. React 하이드레이션 + Zustand 초기화
3. SplashScreen 2.5초 자동 전환 타이머
4. Description 컴포넌트 렌더

이 모든 과정이 끝나야 LCP 요소가 화면에 나타났다.

#### 최적화 Phase 4-5: SSR + SplashScreen 스킵

**Phase 4 — SplashScreen 스킵 (-2.5s)**

온보딩 시작 단계를 `"splash"`에서 `"description"`으로 변경. 2.5초 타이머를 제거하여 LCP에서 2.5초를 직접 절약했다.

**Phase 5 — SSR 온보딩 (LCP = FCP)**

가장 결정적인 최적화. 기존에는 `isLoaded=false` 상태에서 무의미한 skeleton을 SSR했다. 이를 **실제 OnboardingFlow를 SSR하도록 변경**했다:

```typescript
// Before: 서버가 빈 skeleton 렌더 → JS 하이드레이션 후 실제 콘텐츠 → LCP 지연
if (!isLoaded) return <Skeleton />;

// After: 서버가 실제 온보딩 콘텐츠 렌더 → LCP = FCP
if (!isLoaded || !hasCompletedOnboarding) return <OnboardingFlow />;
```

서버와 클라이언트 모두 `isLoaded=false` 상태에서 OnboardingFlow를 렌더하므로 **하이드레이션 불일치가 없다**. Description 컴포넌트의 진입 애니메이션도 초기 마운트 시 `initial={false}`로 비활성화하여, SSR된 콘텐츠가 하이드레이션 후에도 그대로 유지된다.

기존 사용자의 경우: `initialize()`가 localStorage를 동기적으로 읽어 즉시 DailyBoard로 전환하므로 깜빡임이 거의 없다.

|------|--------|-------|------|
| FCP | 6.4s | **2.9s** | **-55%** |
| LCP | 7.3s | **5.0s** | **-32%** |
| TBT | 0ms | 100ms | +100ms |

#### 목표 85+에 도달하지 못한 이유 — 정직한 분석

목표했던 85+에는 도달하지 못했다. 원인을 분석하면:

**남은 병목: framer-motion JS 번들 (~263KiB unused)**

Lighthouse가 지적하는 "Reduce unused JavaScript 263KiB"의 대부분은 framer-motion이다. 이 라이브러리는 카드 애니메이션, 온보딩 트랜지션, 모달 등 앱 전반에 사용되어 쉽게 제거할 수 없다. 모바일 4x CPU throttle 환경에서 이 번들의 파싱 시간이 LCP와 TBT에 직접 영향을 준다.

**구조적 한계: 클라이언트 컴포넌트 아키텍처**

현재 모든 페이지가 `"use client"` 컴포넌트다. 이는 전체 앱을 JS 번들로 보내고 클라이언트에서 하이드레이션해야 함을 의미한다. React Server Components를 전면 도입하면 서버에서 렌더된 HTML이 JS 없이도 인터랙티브할 수 있지만, 이는 앱 아키텍처 전면 재설계를 필요로 한다.

**85+ 달성을 위한 다음 단계:**
- framer-motion → CSS animations 전환 (가장 큰 단일 효과, ~15점 예상)
- React Server Components 도입 (페이지 레벨 RSC + 인터랙티브 부분만 Client)
- 폰트 서브세팅 (197KB → ~50KB)

> **이 과정에서 얻은 핵심 교훈**: 성능 최적화는 "한 번의 마법"이 아니라 **측정 → 가설 → 수정 → 재측정의 반복**이다. 5번의 배포와 5번의 Lighthouse 측정을 거치면서, "폰트가 문제"라는 초기 가설이 틀렸음을(폰트를 고치자 JS 하이드레이션이 드러남) 데이터로 발견하고 방향을 전환한 경험이 가장 값졌다.

### 다시 한다면

**1. i18n 구조를 분리했을 것이다**
카드 데이터에 번역을 인라인한 구조(`titleEn`, `titleJa`, `titleZh`)는 68장 규모에서는 괜찮았다. 하지만 200장으로 확장하면 단일 파일이 수천 줄이 된다. 별도 JSON 번역 파일로 분리하고, 빌드 타임에 합치는 구조가 확장에 유리했을 것이다.

**2. 디자인 탐색에 더 시간을 투자했을 것이다**
2일이라는 제약 때문에 컬러 시스템과 레이아웃을 빠르게 결정했다. 시간이 더 있었다면 라이트 테마 변형, 카드 레이아웃 대안(그리드 vs 스택 vs 팬), 온보딩 플로우의 A/B 테스트까지 진행하고 싶었다.

**3. 병렬 에이전트 분업 경계를 파일 단위로 설정했을 것이다**
JA/ZH 번역 에이전트가 같은 파일을 수정해서 머지 충돌이 발생했다. 처음부터 언어별 별도 파일로 분업했다면 충돌 없이 머지 가능했을 것이다.

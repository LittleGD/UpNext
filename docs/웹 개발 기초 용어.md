# 웹 개발 기초 용어

> 개발 문서를 읽다 보면 자주 나오는 핵심 용어들을 정리했어요.
> 디자인 도구(Figma, Sketch)에 비유해서 설명합니다.

## 프레임워크 (Framework)

**한 줄 요약**: "앱을 만들기 위한 뼈대 + 도구 세트"

Figma로 비유하면, 빈 캔버스에서 시작하는 게 아니라 **Auto Layout, Components, Styles**가 미리 세팅된 디자인 시스템 템플릿을 받는 것과 같아요.

```
라이브러리 vs 프레임워크 차이:

라이브러리 (Library)
→ 내가 필요할 때 꺼내 쓰는 "도구 상자"
→ React, Zustand, Framer Motion
→ Figma 비유: 플러그인

프레임워크 (Framework)
→ 규칙이 있는 "작업 환경" 자체
→ Next.js, Angular
→ Figma 비유: Figma 앱 자체 (레이어, 페이지, 프로토타입 규칙이 정해져 있음)
```

**UpNext에서**: [[Next.js]]가 프레임워크, [[Zustand]]나 [[Framer Motion]]은 라이브러리.

## 페이지 라우팅 (Routing)

**한 줄 요약**: "URL 주소에 따라 어떤 화면을 보여줄지 정하는 시스템"

```
라우팅 = 길 안내

사용자가 주소창에 입력하면:
/              → 홈 (오늘의 카드)
/collection    → 컬렉션 페이지
/settings      → 설정 페이지
```

Figma 비유: 프로토타입에서 **페이지 간 네비게이션 연결**을 만드는 것과 같아요. "이 버튼 누르면 저 페이지로 이동" 하는 것처럼, 라우팅은 "이 URL이면 저 컴포넌트를 보여줘"라고 연결하는 거예요.

### Next.js의 파일 기반 라우팅

Next.js는 특별해요. 코드로 라우팅을 설정하는 대신, **폴더를 만들면 자동으로 URL이 생김**.

```
src/app/
├── page.tsx           →  /
├── collection/
│   └── page.tsx       →  /collection
└── settings/
    └── page.tsx       →  /settings
```

자세한 내용: [[App Router]]

## 렌더링 (Rendering)

**한 줄 요약**: "코드를 사용자가 볼 수 있는 화면(HTML)으로 변환하는 과정"

Figma 비유: 디자인 파일을 **PDF로 내보내기**하는 것과 비슷해요. Figma 안에서는 레이어, 컴포넌트, 오토레이아웃이지만, 내보내면 눈에 보이는 이미지가 되는 것처럼요.

```
React/Next.js 코드 (JSX)     →     렌더링     →     브라우저에 보이는 화면 (HTML)

<ChallengeCard               →     변환      →     <div class="rounded-2xl...">
  card={card}                                         <img .../>
  isSelected={true}                                   <h3>1000보 걷기</h3>
/>                                                  </div>
```

### CSR vs SSR vs SSG

렌더링을 **어디서, 언제** 하느냐에 따라 3가지로 나뉨:

```
┌──────────────────────────────────────────────────────────────┐
│  CSR (Client-Side Rendering) — 클라이언트 사이드 렌더링      │
│  "브라우저가 직접 그린다"                                     │
│                                                              │
│  작동: 빈 HTML 받음 → JS 다운로드 → 브라우저가 화면 그림     │
│  장점: 한번 로드되면 빠른 전환                                │
│  단점: 첫 로딩 느림, SEO 불리                                │
│  예시: UpNext의 "use client" 컴포넌트들                      │
├──────────────────────────────────────────────────────────────┤
│  SSR (Server-Side Rendering) — 서버 사이드 렌더링            │
│  "서버가 미리 그려서 보내준다"                                │
│                                                              │
│  작동: 요청마다 서버가 HTML 완성 → 브라우저에 전달           │
│  장점: SEO 좋음, 첫 화면 빠름                                │
│  단점: 서버 부하, 매 요청마다 처리                            │
│  예시: 실시간 데이터가 필요한 대시보드                        │
├──────────────────────────────────────────────────────────────┤
│  SSG (Static Site Generation) — 정적 사이트 생성             │
│  "빌드할 때 미리 다 그려놓는다"                              │
│                                                              │
│  작동: npm run build 시 HTML 미리 생성 → 그대로 전달         │
│  장점: 가장 빠름, CDN 캐싱 가능                              │
│  단점: 빌드 후 내용 변경 불가 (재빌드 필요)                   │
│  예시: UpNext의 현재 빌드 결과 (○ Static)                    │
└──────────────────────────────────────────────────────────────┘
```

**UpNext에서**: `npm run build` 결과를 보면 모든 페이지가 `○ (Static)` — SSG로 빌드됨. 하지만 실제 동작은 CSR (클라이언트에서 [[Zustand]] 스토어 초기화 후 화면을 그림).

## SEO (Search Engine Optimization)

**한 줄 요약**: "구글 같은 검색엔진이 내 사이트를 잘 찾을 수 있게 최적화하는 것"

```
검색엔진 크롤러(로봇)가 하는 일:
1. 웹사이트 방문
2. HTML 읽기
3. "이 사이트는 뭐에 대한 거지?" 판단
4. 검색 결과에 반영

CSR 문제: 크롤러가 왔을 때 빈 HTML만 있음 → "내용 없음" 판단
SSR/SSG 장점: 이미 완성된 HTML → 크롤러가 바로 내용 파악 가능
```

### Next.js에서의 SEO 설정

```typescript
// src/app/layout.tsx에 있는 코드
export const metadata: Metadata = {
  title: "UpNext",
  description: "매일 작은 성취를 쌓아가는 로그라이크 챌린지",
};
```

이 코드가 HTML `<head>`에 들어가서 검색엔진에게 "이 사이트는 챌린지 앱이야"라고 알려줌.

**UpNext에서**: 현재는 PWA(모바일 앱)를 목표로 하므로 SEO가 핵심은 아니지만, Next.js가 기본 지원해줌.

## 빌드 (Build)

**한 줄 요약**: "개발용 코드를 실제 배포용 코드로 변환하는 과정"

```
개발 모드 (npm run dev)          배포 모드 (npm run build)
─────────────────────            ─────────────────────
- 읽기 쉬운 코드                 - 압축된 코드
- 에러 메시지 상세               - 최적화됨
- 느림 (실시간 변환)              - 빠름 (미리 처리)
- 내 컴퓨터에서만                 - 어디서든 실행 가능
```

Figma 비유: 작업 중인 .fig 파일 vs 최종 납품용 에셋 내보내기.

## 컴포넌트 트리 (Component Tree)

**한 줄 요약**: "컴포넌트가 부모-자식 관계로 중첩된 구조"

```
UpNext 컴포넌트 트리:

RootLayout                    ← 최상위 (HTML, 폰트, 다크 테마)
├── Header                    ← 상단 바
├── main                      ← 페이지 내용
│   └── Home (page.tsx)
│       ├── OnboardingFlow    ← 온보딩 (처음 1회)
│       │   ├── SplashScreen
│       │   ├── AppDescription
│       │   ├── DifficultySelect
│       │   └── StarterPackSelect
│       ├── CardDrawScreen    ← 카드 뽑기
│       │   └── FanCard (×6)
│       └── DailyBoard        ← 오늘의 챌린지
│           └── ChallengeCard (×N)
└── BottomNav                 ← 하단 네비게이션
```

Figma 비유: **레이어 패널**의 구조와 동일. Frame 안에 Frame이 있고, 그 안에 컴포넌트가 있는 것처럼, React도 컴포넌트 안에 컴포넌트가 중첩됨.

자세한 내용: [[React 컴포넌트]], [[Props]]

## 디자인 토큰 (Design Tokens)

**한 줄 요약**: "색상, 폰트 등 디자인 값을 변수로 관리하는 시스템"

이건 디자이너에게 익숙한 개념! Figma의 **Variables/Styles**와 동일.

```css
/* CSS 커스텀 프로퍼티 = 디자인 토큰 */
:root {
  --bg-primary: #0A0A0A;        /* Figma: Color/bg-primary */
  --accent-primary: #00FF41;    /* Figma: Color/accent-primary */
  --font-display: 'April16Promise';  /* Figma: Typography/Display */
}
```

**왜 중요한가**: 색상 하나를 바꾸면 **모든 컴포넌트에 자동 반영**. Figma에서 Style을 수정하면 전체 파일에 적용되는 것과 같은 원리.

**UpNext에서**: `globals.css`의 `:root` 블록이 디자인 토큰, `@theme inline` 블록이 [[Tailwind CSS]]에 등록하는 부분.
